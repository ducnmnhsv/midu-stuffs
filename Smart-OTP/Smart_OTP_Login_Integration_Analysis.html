❓ Câu trả lời cho 2 câu hỏi của bạn

1. Có cần thêm grant_type mới không? Không. Smart OTP vẫn dùng grant_type: "password_otp" hiện tại. Chỉ cần mở rộng Step 2 (verifyOTP) để nhận thêm param otpType: "SMART_OTP". Backend tự route sang Smart OTP verification thay vì hash lookup SMS.
Tại sao không cần grant_type mới? grant_type phân biệt cơ chế xác thực (password vs biometric vs social). Smart OTP chỉ thay đổi phương thức xác nhận OTP — vẫn là password + OTP, chỉ OTP source khác. Tương tự SMS OTP vs Push OTP — cùng grant_type, khác delivery method.
2. Flow user đã có vs chưa có Smart OTP khác nhau thế nào? Khác nhau ở Step 2 (màn hình nhập OTP). Step 1 (nhập username/password) hoàn toàn giống nhau. Backend trả về sotpStatus trong response Step 1 → FE tự biết show UI nào.

🔄 Luồng Đăng Nhập Hiện Tại (Baseline)

Đây là flow đang chạy production. Smart OTP sẽ được tích hợp vào đây.

1
User nhập username + password
POST/api/v1/login — grant_type: "password_otp"
↓
2
Backend xác thực password → Lotte trả về OTP hash
Response: accessToken (TẠM THỜI) + mfaData (OTP đã mã hóa RSA)
↓
3
Backend gửi OTP tới user
OneSignal push notification hoặc SMS (fallback)
↓
4
User nhập mã OTP nhận được
POST/api/v1/login/sec/verifyOTP — { otpValue: "1234" }
↓
✓
Login thành công
Response: accessToken (FINAL) + refreshToken + userInfo
Vấn đề hiện tại với SMS/Push OTP: OTP hiện tại hiển thị qua push notification → có thể bị nhìn trộm trên màn hình khóa. Smart OTP giải quyết vấn đề này bằng cách để user tự gen mã trong app, không cần backend gửi.

📊 3 Cases Cần Handle

Case A — Đã kích hoạt Smart OTP

sotpStatus = "Y"

  • 🔐Nhập username + password → Submit
  • 📡Step 1 response: otpType: "SMART_OTP"
  • 📱FE show: "Mở app → nhập PIN → lấy mã Smart OTP"
  • ⌨️User nhập mã 6 chữ số từ app
  • ✅verifyOTP với otpType: "SMART_OTP"
  • 🎉Login thành công
UX improvement: Không cần chờ push/SMS. User chủ động lấy mã.
Case B — Chưa kích hoạt Smart OTP

sotpStatus = "N"

  • 🔐Nhập username + password → Submit
  • 📡Step 1 response: otpType: "PUSH_OTP"
  • 📲FE show: "Nhập mã OTP vừa gửi qua app/SMS"
  • ⌨️User nhập OTP nhận được (flow hiện tại, không đổi)
  • ✅Login thành công
  • 💬[Optional] Banner: "Kích hoạt Smart OTP để bảo mật hơn?"
Backward compatible: Flow cũ hoàn toàn không thay đổi.
Case C — Edge Case: Mất device / App bị xóa

sotpStatus = "Y" nhưng user không gen được mã (app bị xóa, đổi điện thoại…)

  • ⚠️Step 1 trả về otpType: "SMART_OTP"
  • 🔄FE cần có link "Không lấy được mã?"
  • 📱User chọn → fallback về SMS OTP
  • ✅verifyOTP với otpType: "SMS_OTP"
Yêu cầu kỹ thuật: Backend phải accept otpType: "SMS_OTP" ngay cả khi sotpStatus = "Y". Cần rate limit riêng để tránh abuse. Sau login thành công qua SMS fallback → gợi ý user kích hoạt lại Smart OTP.

🔧 Thay Đổi API Cần Thiết (Minimal)

Chỉ 2 thay đổi nhỏ — không breaking change, không grant_type mới.

Thay đổi 1: Step 1 Response — Thêm otpType

TRƯỚC (hiện tại)
POST /api/v1/login
Response:
{
  "accessToken": "temp_token",
  "otpIndex": "1",
  "userData": {
    "mfaData": "RSA_encrypted..."
  }
}
SAU (thêm 2 fields)
POST /api/v1/login
Response:
{
  "accessToken": "temp_token",
  "otpIndex": "1",
  "userData": {
    "mfaData": "RSA_encrypted..."
  },
  "sotpStatus": "Y" | "N",
  "otpType": "SMART_OTP" | "SMS_OTP"
}

Thay đổi 2: Step 2 Request — Thêm otpType

TRƯỚC (hiện tại)
POST /api/v1/login/sec/verifyOTP
Body:
{
  "otpValue": "1234"
}
SAU (thêm 1 optional field)
POST /api/v1/login/sec/verifyOTP
Body:
{
  "otpValue": "123456",
  "otpType": "SMART_OTP" | "SMS_OTP" // optional, default: SMS_OTP
}
Backend routing logic (verifyOTP):
Nếu otpType == "SMART_OTP" → gọi nội bộ /api/v1/smartOtp/verify (đã có)
Nếu otpType == "SMS_OTP" hoặc không có → dùng flow hiện tại (apache-crypt hash lookup)
