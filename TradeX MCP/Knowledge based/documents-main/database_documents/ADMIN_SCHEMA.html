Admin & User Management Database Schema

This document describes the database schema for the Admin and User Management domain.

Overview

The Admin domain manages system users, brokers, authorities/roles, chat rooms, system configuration, and administrative functions across the NHSV platform.

Entity Relationship Diagram

Field Descriptions

t_user

  • id: Primary key identifier for the user
  • login: Unique login username
  • password_hash: Hashed password
  • full_name: User’s full name
  • email: User’s email address
  • image_url: URL to user’s profile image
  • activated: Flag indicating if the account is activated
  • lang_key: Language preference key
  • activation_key: Key used for account activation
  • reset_key: Key used for password reset
  • created_date: Timestamp when the account was created
  • reset_date: Timestamp when password was reset
  • created_by: User who created this account
  • last_modified_by: User who last modified this account
  • last_modified_date: Timestamp when account was last modified
  • deactivated_at: Timestamp when account was deactivated
  • deactivated_by: User who deactivated this account

jhi_user

  • id: Primary key identifier for the user (JHipster)
  • login: Unique login username
  • password_hash: Hashed password
  • first_name: User’s first name
  • last_name: User’s last name
  • email: User’s email address
  • activated: Flag indicating if the account is activated
  • lang_key: Language preference key
  • image_url: URL to user’s profile image
  • activation_key: Key used for account activation
  • reset_key: Key used for password reset
  • reset_date: Timestamp when password was reset
  • created_date: Timestamp when the account was created
  • created_by: User who created this account
  • last_modified_date: Timestamp when account was last modified
  • last_modified_by: User who last modified this account

t_authority

  • name: Primary key - authority/role name
  • description: Description of the authority/role

jhi_authority

  • name: Primary key - authority/role name (JHipster)

t_user_authority

  • user_id: Foreign key to the user
  • authority_name: Foreign key to the authority/role

t_broker

  • id: Primary key identifier for the broker
  • username: Broker’s username
  • fullname: Broker’s full name
  • status: Broker status (active, inactive, etc.)
  • total_chat_room: Total number of chat rooms created
  • total_viewed_chat_room: Total number of chat rooms viewed
  • current_rank: Current ranking position
  • is_dynamic: Flag indicating if ranking is dynamic
  • email: Broker’s email address
  • created_at: Timestamp when the broker was created
  • updated_at: Timestamp when the broker was last updated
  • deactivated_at: Timestamp when the broker was deactivated
  • deactivated_by: User who deactivated this broker
  • invited_by: User who invited this broker
  • photo: URL to broker’s photo
  • introduction: Broker’s introduction/bio

t_invite_user

  • id: Primary key identifier for the invitation
  • email: Email address of the invited user
  • invite_code: Unique invitation code
  • status: Invitation status (PENDING, ACCEPTED, REJECTED, EXPIRED)
  • invited_by: Foreign key to the user who sent the invitation
  • invited_at: Timestamp when the invitation was sent
  • accepted_at: Timestamp when the invitation was accepted
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

t_chat_room

  • id: Primary key identifier for the chat room
  • room_name: Name of the chat room
  • room_type: Type of chat room (PUBLIC, PRIVATE, etc.)
  • broker_id: Foreign key to the broker who hosts this room
  • description: Description of the chat room
  • max_participants: Maximum number of participants allowed
  • current_participants: Current number of participants
  • is_active: Flag indicating if the chat room is active
  • created_at: Timestamp when the chat room was created
  • updated_at: Timestamp when the chat room was last updated

t_created_chat_room

  • id: Primary key identifier for the creation record
  • chat_room_id: Foreign key to the chat room
  • created_by: Foreign key to the user who created the room
  • created_at: Timestamp when the room was created

t_recent_view_chat_room

  • id: Primary key identifier for the view record
  • chat_room_id: Foreign key to the chat room
  • user_id: Foreign key to the user who viewed the room
  • last_viewed_at: Timestamp when the room was last viewed
  • created_at: Timestamp when the record was created

t_social_link

  • id: Primary key identifier for the social link
  • broker_id: Foreign key to the broker
  • platform: Social media platform (FACEBOOK, TWITTER, YOUTUBE, etc.)
  • link_url: URL to the social media profile/page
  • is_active: Flag indicating if the link is active
  • created_at: Timestamp when the link was created
  • updated_at: Timestamp when the link was last updated

t_feedback

  • id: Primary key identifier for the feedback
  • user_id: Foreign key to the user who submitted feedback
  • feedback_type: Type of feedback (BUG, FEATURE_REQUEST, COMPLAINT, etc.)
  • title: Title of the feedback
  • content: Content/details of the feedback
  • status: Feedback status (PENDING, REVIEWED, RESOLVED, etc.)
  • created_at: Timestamp when the feedback was created
  • updated_at: Timestamp when the feedback was last updated

t_stock_event

  • id: Primary key identifier for the stock event
  • event_type: Type of event (DIVIDEND, SPLIT, MERGER, etc.)
  • symbol: Stock symbol
  • description: Description of the event
  • event_date: Date when the event occurs
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

t_holiday

  • id: Primary key identifier for the holiday
  • holiday_name: Name of the holiday
  • holiday_date: Date of the holiday
  • holiday_type: Type of holiday (NATIONAL, TRADING_HOLIDAY, etc.)
  • is_active: Flag indicating if the holiday is active
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

market_history_job_result

  • id: Primary key identifier for the job result
  • job_name: Name of the background job
  • status: Job status (RUNNING, COMPLETED, FAILED, etc.)
  • result_data: Job result data in JSON format
  • started_at: Timestamp when the job started
  • completed_at: Timestamp when the job completed
  • created_at: Timestamp when the record was created

Core Entities

t_user / jhi_user

System users. t_user is used in nhsv-admin and copy-trading-engine, while jhi_user is used in ekyc-admin (JHipster-based).

Key Fields: - login: Username/login identifier - password_hash: Hashed password - email: User email address - activated: Account activation status - full_name / first_name, last_name: User’s name

Key Relationships: - One-to-many with t_user_authority (has roles) - One-to-one with t_broker (can be a broker) - One-to-many with t_invite_user (can invite users) - One-to-many with t_chat_room (participates in chat) - One-to-many with t_feedback (submits feedback)

t_authority / jhi_authority

User roles/authorities. Defines what permissions users have.

Key Relationships: - Many-to-many with t_user via t_user_authority (assigned to users)

t_user_authority

Junction table mapping users to their authorities/roles.

Key Relationships: - Many-to-one with t_user (belongs to user) - Many-to-one with t_authority (belongs to authority)

t_broker

Broker/market leader information. Brokers are special users who can host chat rooms and be followed in copy trading.

Key Fields: - current_rank: Broker’s current ranking - total_chat_room: Number of chat rooms created - total_viewed_chat_room: Number of chat rooms viewed - is_dynamic: Whether broker ranking is dynamic - photo, introduction: Broker profile information

Key Relationships: - One-to-one with t_user (is a user) - One-to-many with t_chat_room (hosts chat rooms) - One-to-many with t_social_link (has social media links)

t_invite_user

User invitation system. Tracks invitations sent to new users.

Key Relationships: - Many-to-one with t_user (invited by user)

t_chat_room

Chat rooms where users can communicate. Brokers can host chat rooms.

Key Fields: - room_type: Type of chat room - broker_id: Broker hosting the room - max_participants: Maximum number of participants - current_participants: Current number of participants

Key Relationships: - Many-to-one with t_broker (hosted by broker) - One-to-many with t_created_chat_room (creation tracking) - One-to-many with t_recent_view_chat_room (view tracking)

t_created_chat_room

Tracks who created which chat rooms.

Key Relationships: - Many-to-one with t_chat_room (belongs to chat room) - Many-to-one with t_user (created by user)

t_recent_view_chat_room

Tracks recently viewed chat rooms for each user.

Key Relationships: - Many-to-one with t_chat_room (belongs to chat room) - Many-to-one with t_user (viewed by user)

t_social_link

Social media links for brokers.

Key Relationships: - Many-to-one with t_broker (belongs to broker)

t_feedback

User feedback and support tickets.

Key Relationships: - Many-to-one with t_user (submitted by user)

Supporting Entities

t_stock_event

Stock market events and announcements.

Usage: Tracks important events related to stocks.

t_holiday

Trading holidays and market closures.

Usage: Used to determine when markets are closed.

market_history_job_result

Results of background jobs that process market history data.

Usage: Tracks execution and results of market data processing jobs.

User Management Flow

User Registration → Activation → Role Assignment → Full Access
  1. Registration: New user account created (t_user)
  2. Activation: User activates account (updates activated field)
  3. Role Assignment: Authorities assigned via t_user_authority
  4. Full Access: User can access features based on roles

Broker Management Flow

User → Broker Profile → Chat Rooms → Social Links
  1. User becomes a broker (creates t_broker record)
  2. Broker can create chat rooms (t_chat_room)
  3. Broker can add social links (t_social_link)
  4. Broker ranking and statistics are tracked

Chat Room Flow

Broker → Create Chat Room → Users Join → Track Views
  1. Broker creates a chat room (t_chat_room)
  2. Creation is tracked (t_created_chat_room)
  3. Users join and view the room
  4. Views are tracked (t_recent_view_chat_room)

Authority/Role System

Authority Definition → User Assignment → Permission Check
  1. Authorities are defined (t_authority)
  2. Users are assigned authorities (t_user_authority)
  3. System checks user authorities for permission control

Invitation System

User → Send Invitation → Invitee Accepts → New User Created
  1. User sends invitation (t_invite_user)
  2. Invitee receives invitation code
  3. Invitee accepts and creates account
  4. Invitation status is updated

Notes

  • t_user and jhi_user serve similar purposes but are in different repositories
  • Brokers are a special type of user with additional capabilities
  • Chat rooms are primarily broker-hosted
  • The authority system provides role-based access control
  • All entities include audit fields for tracking changes
