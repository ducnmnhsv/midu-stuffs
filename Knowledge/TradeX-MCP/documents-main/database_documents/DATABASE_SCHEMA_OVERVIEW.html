Schema Overview

The NHSV database consists of 75 tables organized across multiple functional domains:

  • AAA Domain: 23 tables - Authentication, authorization, OAuth2, and partner integration
  • Configuration Domain: 24 tables - System configuration, language resources, menus, FAQs, and OAuth2 settings
  • Copy Trading Domain: 20+ tables - Copy trading functionality, portfolios, subscribers, and orders
  • eKYC Domain: 14 tables - Electronic KYC processing and contract management
  • Admin Domain: 20+ tables - User management, brokers, chat rooms, and system configuration
  • Notification Domain: 2 tables - SMS and notification management
  • VietStock Bridge Domain: 2 tables - Stock news and stock-news relationships

Domain Schemas

AAA (Authentication, Authorization, and Accounting)

The AAA domain handles all authentication, authorization, OAuth2 flows, and partner integrations.

Core Entities: - t_service_user - Service user accounts - t_client - OAuth2 clients - t_refresh_token - OAuth2 refresh tokens - t_biometric - Biometric authentication data - t_otp - One-time passwords - t_transaction - Transaction logs for API calls

Authorization Entities: - t_scope - OAuth2 scopes - t_scope_group - Scope groups - t_scope_scope_group_map - Scope to scope group mapping - t_login_method - Login methods configuration - t_login_method_step - Multi-step login process - t_client_login_method_map - Client to login method mapping - t_login_method_scope_group_map - Login method to scope group mapping - t_login_method_step_scope_group_map - Login step to scope group mapping

Partner Integration: - t_partner - Partner configurations - t_link_account - Linked partner accounts - t_link_account_draft - Draft link account requests - t_link_account_history - Link account history

Supporting Entities: - t_username_mapping - Username mappings - t_order_pass - Order pass management - t_access_token_history - Access token history - t_service_monitor - Service monitoring - t_account_demo - Demo accounts

See AAA_SCHEMA.md for detailed ER diagrams.

Configuration

The Configuration domain manages system-wide configuration including language resources, menus, FAQs, OAuth2 scopes, login methods, and service configurations.

Core Entities: - t_menu, t_menu_group, t_menu_role, t_admin_role - Menu management system - t_faq, t_faq_group, t_faq_review - FAQ management - t_lang_namespace, t_lang_key, t_lang_translate - Language resource management - t_lang_resource, t_lang_resource_version, t_lang_resource_file - Language resource files - t_scope, t_scope_group - OAuth2 scope management - t_client, t_login_method, t_login_method_step - OAuth2 and authentication configuration - t_data_view, t_service, t_holiday, t_interest_info - System configuration - t_open_api, t_template_resource - API and template management

See CONFIGURATION_SCHEMA.md for detailed ER diagrams.

Copy Trading

The Copy Trading domain manages copy trading functionality including portfolios, subscribers, orders, and market leader tracking.

Core Entities: - t_copy_portfolio - Copy trading portfolios - t_copy_portfolio_details - Portfolio details - t_copy_portfolio_history - Portfolio history - t_copy_portfolio_detail_history - Portfolio detail history

Subscriber Management: - t_copy_subscriber - Copy trading subscribers - t_copy_subscriber_details - Subscriber details - t_copy_subscriber_history - Subscriber history

Trading: - t_copy_trading_order - Copy trading orders - t_copy_trading_register - Copy trading registrations

Market Leader: - t_copy_market_leader_details - Market leader details - t_copy_market_leader_profit_loss - Market leader profit/loss - t_copy_market_leader_profit_loss_details - Market leader profit/loss details

See COPY_TRADING_SCHEMA.md for detailed ER diagrams.

eKYC (Electronic Know Your Customer)

The eKYC domain handles electronic KYC processing, contract management, and customer verification.

Core Entities: - e_kyc - Main eKYC records - ekyc_ext - Extended eKYC information - ekyc_additional_info - Additional eKYC information - ekyc_creator_status - eKYC creator status - ekyc_bank_list - Bank list for eKYC

Contract Management: - econtract - Electronic contracts - econtract_info - Contract information - econtract_info_history - Contract information history

Supporting Entities: - blockholder - Blockholder information - public_coop - Public cooperation data - matching_rate - Matching rate data - ttl_issue_place_code_map - Issue place code mapping

See EKYC_SCHEMA.md for detailed ER diagrams.

Admin & User Management

The Admin domain manages users, brokers, chat rooms, and system configuration.

User Management: - t_user - System users (multiple repositories) - jhi_user - JHipster users (ekyc-admin) - t_authority - User authorities/roles - jhi_authority - JHipster authorities - t_user_authority - User to authority mapping

Broker Management: - t_broker - Broker information - t_invite_user - Invited users - t_social_link - Social links

Chat & Communication: - t_chat_room - Chat rooms - t_created_chat_room - Created chat rooms - t_recent_view_chat_room - Recently viewed chat rooms

System Configuration: - t_stock_event - Stock events - t_holiday - Holidays - t_feedback - User feedback - market_history_job_result - Market history job results

See ADMIN_SCHEMA.md for detailed ER diagrams.

Notification

The Notification domain handles SMS and notification management.

Entities: - t_notification_list - Notification list - t_fpt_sms_history - FPT SMS history

See NOTIFICATION_SCHEMA.md for detailed ER diagrams.

VietStock Bridge

The VietStock Bridge domain provides integration with VietStock data sources for stock market information and news.

Core Entities: - t_stock_news - Stock news articles with multi-language support - t_stock_stock_news - Junction table linking news to stock symbols

Supporting Data: - MongoDB collections for symbol info, financial statements, and watchlists

See VIETSTOCK_BRIDGE_SCHEMA.md for detailed ER diagrams.

Notes

  • Most entities use DefaultBaseModel which includes audit fields: created_by, updated_by, created_at, updated_at
  • Foreign key relationships are primarily managed through application logic rather than database constraints
  • Some tables exist in multiple repositories (e.g., t_user appears in both copy-trading-engine and nhsv-admin)
  • The aaa repository uses a custom ORM with TypeScript, while other repositories use JPA/Hibernate
