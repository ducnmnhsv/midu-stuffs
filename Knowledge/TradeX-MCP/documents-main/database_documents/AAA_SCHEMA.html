AAA (Authentication, Authorization, and Accounting) Database Schema

This document describes the database schema for the AAA (Authentication, Authorization, and Accounting) domain.

Overview

The AAA domain provides OAuth2-based authentication, fine-grained authorization through scopes, and transaction logging. It also handles partner integrations and biometric authentication.

Entity Relationship Diagram

Field Descriptions

t_service_user

  • id: Primary key identifier for the service user
  • username: Unique username for the service user account
  • email: Email address of the service user
  • phone_code: Country/region phone code (e.g., +84 for Vietnam)
  • phone_number: Phone number without country code
  • birthday: User’s date of birth
  • avatar: URL or path to user’s profile picture
  • register_mobile_otp: Flag indicating if mobile OTP registration is enabled
  • created_by: User ID who created this record
  • updated_by: User ID who last updated this record
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

t_client

  • id: Primary key identifier for the OAuth2 client
  • user_id: Foreign key to the service user who owns this client
  • client_id: OAuth2 client identifier (public)
  • client_secret: OAuth2 client secret (private, used for authentication)
  • description: Human-readable description of the client application
  • status: Client status (active, inactive, etc.)
  • app_version: Version of the client application
  • created_by: User ID who created this record
  • updated_by: User ID who last updated this record
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

t_refresh_token

  • id: Primary key identifier for the refresh token
  • client_id: Foreign key to the OAuth2 client that issued this token
  • user_id: Foreign key to the user account
  • service_user_id: Foreign key to the service user account
  • login_method_id: Foreign key to the login method used
  • token: The refresh token string
  • updated_at: Timestamp when the token was last updated/refreshed
  • extend_data: JSON string containing extended token metadata
  • source_ip: IP address from which the token was issued
  • device_type: Type of device (mobile, web, desktop, etc.)
  • expired_at: Timestamp when the token expires
  • parent_id: Reference to parent token (for token rotation)
  • mac_address: MAC address of the device
  • platform: Platform identifier (iOS, Android, Web, etc.)
  • os_version: Operating system version
  • app_version: Application version

t_biometric

  • id: Primary key identifier for the biometric record
  • user_id: Foreign key to the user account
  • password: Encrypted biometric password/template
  • public_key: Public key for biometric authentication
  • username: Username associated with this biometric
  • grant_type: OAuth2 grant type used
  • client_id: Foreign key to the OAuth2 client
  • platform: Platform identifier (iOS, Android, etc.)
  • os_version: Operating system version
  • app_version: Application version
  • device_id: Unique device identifier
  • source_ip: IP address from which biometric was registered
  • is_deleted: Flag indicating if the biometric record is deleted
  • delete_reason: Reason for deletion
  • created_by: User ID who created this record
  • updated_by: User ID who last updated this record
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

t_otp

  • id: Primary key identifier for the OTP record
  • user_id: Foreign key to the user account
  • service_user_id: Foreign key to the service user account
  • client_id: Foreign key to the OAuth2 client
  • login_method_id: Foreign key to the login method
  • refresh_token_id: Foreign key to the refresh token
  • otp_code: The one-time password code
  • otp_type: Type of OTP (SMS, Email, etc.)
  • expired_at: Timestamp when the OTP expires
  • is_used: Flag indicating if the OTP has been used
  • source_ip: IP address from which OTP was requested
  • device_type: Type of device
  • created_at: Timestamp when the OTP was created

t_transaction

  • id: Primary key identifier for the transaction log
  • client_id: Foreign key to the OAuth2 client
  • user_id: Foreign key to the user account
  • service_user_id: Foreign key to the service user account
  • refresh_token_id: Foreign key to the refresh token used
  • scope_id: Foreign key to the scope used in this transaction
  • request_method: HTTP method (GET, POST, etc.)
  • request_url: URL of the API endpoint accessed
  • request_body: Request body content
  • response_code: HTTP response code
  • response_body: Response body content
  • source_ip: IP address from which the request originated
  • user_agent: User agent string from the request
  • created_at: Timestamp when the transaction occurred

t_scope

  • id: Primary key identifier for the scope
  • name: Unique scope name (e.g., “read:profile”, “write:orders”)
  • uri_pattern: URI pattern that this scope grants access to
  • forward_type: Type of forwarding mechanism
  • forward_data: JSON data for forwarding configuration
  • created_by: User ID who created this record
  • updated_by: User ID who last updated this record
  • created_at: Timestamp when the record was created
  • updated_at: Timestamp when the record was last updated

t_scope_group

Core Entities

t_service_user

The central user entity for the AAA system. Stores user account information including username, email, phone, and avatar.

Key Relationships: - One-to-many with t_client (users can have multiple OAuth2 clients) - One-to-many with t_refresh_token (users can have multiple active sessions) - One-to-many with t_biometric (users can have multiple biometric authentications) - One-to-many with t_link_account (users can link multiple partner accounts) - One-to-many with t_transaction (users perform multiple transactions)

t_client

OAuth2 client applications. Each client has a client_id and client_secret for authentication.

Key Relationships: - Many-to-one with t_service_user (belongs to a user) - One-to-many with t_refresh_token (issues tokens) - Many-to-many with t_login_method (can use multiple login methods) - One-to-many with t_transaction (initiates transactions)

t_refresh_token

OAuth2 refresh tokens for maintaining user sessions. Contains device and platform information.

Key Relationships: - Many-to-one with t_client (issued by client) - Many-to-one with t_service_user (belongs to user) - One-to-many with t_transaction (authorizes transactions) - One-to-many with t_otp (can generate OTPs) - One-to-many with t_access_token_history (tracks access token history)

t_scope

OAuth2 scopes define what resources a client can access. Scopes are grouped into scope groups for easier management.

Key Relationships: - Many-to-many with t_scope_group (scopes belong to groups) - One-to-many with t_transaction (used in transactions)

t_login_method

Defines different login methods (e.g., password, OTP, biometric) and their configuration.

Key Relationships: - One-to-many with t_login_method_step (multi-step login processes) - Many-to-many with t_scope_group (grants scope groups) - Many-to-many with t_client (used by clients)

t_partner

External partner integrations (e.g., trading platforms). Stores partner configuration and API endpoints.

Key Relationships: - One-to-many with t_link_account (provides linked accounts)

t_link_account

Links user accounts with partner accounts, enabling cross-platform functionality.

Key Relationships: - Many-to-one with t_partner (belongs to a partner) - Many-to-one with t_service_user (belongs to a user) - One-to-many with t_link_account_history (tracks changes)

Authorization Flow

User → Client → Login Method → Scope Groups → Scopes → Resources
  1. User authenticates via a Client
  2. Client uses a Login Method
  3. Login Method grants Scope Groups
  4. Scope Groups contain Scopes
  5. Scopes authorize access to specific resources

Authentication Flow

Service User → Biometric/OTP → Refresh Token → Access Token → Transaction
  1. User authenticates (via biometric, OTP, or password)
  2. System issues a Refresh Token
  3. Refresh Token is used to generate Access Tokens
  4. Access Tokens authorize API transactions
  5. Transactions are logged in t_transaction

Partner Integration Flow

Partner → Link Account Draft → Link Account → Link Account History
  1. Partner configuration is stored in t_partner
  2. User initiates link via t_link_account_draft
  3. Upon confirmation, draft becomes t_link_account
  4. All changes are tracked in t_link_account_history
