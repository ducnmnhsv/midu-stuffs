Notification Database Schema

This document describes the database schema for the Notification domain.

Overview

The Notification domain handles SMS notifications, email notifications, and general notification management for the NHSV platform.

Entity Relationship Diagram

Field Descriptions

t_notification_list

  • id: Primary key identifier for the notification
  • notification_type: Type of notification (SMS, EMAIL, PUSH, IN_APP)
  • recipient_type: Type of recipient (USER, BROKER, SYSTEM, etc.)
  • recipient_id: ID of the recipient
  • recipient_phone: Phone number for SMS notifications
  • recipient_email: Email address for email notifications
  • title: Notification title/subject
  • content: Notification content/message body
  • template_id: Template identifier used for the notification
  • status: Current status (PENDING, SENT, FAILED, DELIVERED, CANCELLED)
  • retry_count: Number of retry attempts made
  • scheduled_at: Timestamp when notification should be sent (for scheduled notifications)
  • sent_at: Timestamp when notification was actually sent
  • created_at: Timestamp when the notification was created
  • updated_at: Timestamp when the notification was last updated

t_fpt_sms_history

  • id: Primary key identifier for the SMS history record
  • notification_id: Foreign key to the notification in t_notification_list
  • phone_number: Recipient phone number
  • message: SMS message content
  • status: SMS status (SENT, DELIVERED, FAILED, PENDING)
  • response_code: Response code from FPT SMS gateway
  • response_message: Response message from FPT SMS gateway
  • sent_at: Timestamp when SMS was sent to the gateway
  • delivered_at: Timestamp when SMS was delivered to recipient (if applicable)
  • created_at: Timestamp when the record was created

Core Entities

t_notification_list

Main notification queue and management table. Stores all notifications to be sent or that have been sent.

Key Fields: - notification_type: Type of notification (SMS, EMAIL, PUSH, etc.) - recipient_type: Type of recipient (USER, BROKER, SYSTEM, etc.) - recipient_id: ID of the recipient - recipient_phone: Phone number for SMS notifications - recipient_email: Email address for email notifications - title: Notification title - content: Notification content/message - template_id: Template used for the notification - status: Current status (PENDING, SENT, FAILED, DELIVERED) - retry_count: Number of retry attempts - scheduled_at: When notification should be sent - sent_at: When notification was actually sent

Key Relationships: - One-to-many with t_fpt_sms_history (SMS notifications tracked)

t_fpt_sms_history

SMS notification history specifically for FPT SMS service. Tracks all SMS messages sent through the FPT gateway.

Key Fields: - phone_number: Recipient phone number - message: SMS message content - status: SMS status (SENT, DELIVERED, FAILED) - response_code: Response code from SMS gateway - response_message: Response message from SMS gateway - sent_at: When SMS was sent - delivered_at: When SMS was delivered (if applicable)

Key Relationships: - Many-to-one with t_notification_list (belongs to notification)

Notification Flow

Notification Request → Queue → Process → Send → Track
  1. Notification Request: System creates notification record (t_notification_list)
  2. Queue: Notification is queued with status = PENDING
  3. Process: Notification service processes the queue
  4. Send:
    • For SMS: Sends via FPT gateway, creates t_fpt_sms_history record
    • For Email: Sends via email service
    • For Push: Sends via push notification service
  5. Track: Status updated, delivery tracked

SMS Notification Flow

Notification → FPT SMS Gateway → Delivery Status → History
  1. Notification is created with notification_type = SMS
  2. SMS is sent via FPT gateway
  3. FPT SMS history record is created (t_fpt_sms_history)
  4. Delivery status is updated based on gateway response
  5. History is maintained for audit and tracking

Notification Types

  • SMS: Text messages sent to phone numbers
  • EMAIL: Email notifications
  • PUSH: Push notifications (mobile apps)
  • IN_APP: In-app notifications

Status Values

t_notification_list.status

  • PENDING: Waiting to be sent
  • SENT: Successfully sent
  • FAILED: Failed to send
  • DELIVERED: Delivered to recipient
  • CANCELLED: Cancelled before sending

t_fpt_sms_history.status

  • SENT: SMS sent to gateway
  • DELIVERED: SMS delivered to recipient
  • FAILED: SMS failed to send
  • PENDING: Waiting for delivery confirmation

Retry Mechanism

Notifications can be retried if they fail: - retry_count tracks number of retry attempts - System can retry failed notifications based on configuration - Maximum retry count can be set per notification type

Template System

Notifications can use templates: - template_id references notification templates - Templates define message format and content - Supports variable substitution for personalized messages

Integration Points

FPT SMS Gateway

  • t_fpt_sms_history integrates with FPT SMS service
  • Tracks response codes and messages from gateway
  • Monitors delivery status

Email Service

  • Email notifications are sent via email service
  • Delivery tracking can be added if needed

Push Notification Service

  • Push notifications for mobile apps
  • Can be extended with delivery tracking

Audit and Tracking

  • All notifications are logged in t_notification_list
  • SMS notifications have detailed history in t_fpt_sms_history
  • Timestamps track creation, scheduling, sending, and delivery
  • Status fields provide visibility into notification lifecycle

Notes

  • The notification system is designed to be extensible
  • Additional notification types can be added
  • SMS history is specifically tracked for FPT gateway
  • Email and push notification tracking can be extended similarly
  • The system supports scheduled notifications via scheduled_at field
