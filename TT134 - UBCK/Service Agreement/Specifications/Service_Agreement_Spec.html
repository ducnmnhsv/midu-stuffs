Bối cảnh & Yêu cầu pháp lý

Khoản 4 Điều 5 TT134 yêu cầu tổ chức cung cấp dịch vụ phải thể hiện dịch vụ bằng hợp đồng hoặc điều khoản hợp đồng với khách hàng, bao gồm 3 nội dung bắt buộc.

#Yêu cầuMô tảIT Impact
1 Phương thức GD & loại GD Hợp đồng phải nêu rõ phương thức giao dịch trực tuyến (Mobile App, Web Trading, API) và loại giao dịch tương ứng (lệnh thường, lệnh dừng, TP/SL, OCO) BEFE
2 SĐT đăng ký + xác minh Số điện thoại di động đăng ký phải được xác minh thuộc quyền sử dụng hợp pháp của KH. Cần biện pháp xác minh (OTP + cross-check thông tin cá nhân) BEFE
3 Rủi ro & trách nhiệm Hợp đồng phải nêu rõ các rủi ro và trách nhiệm bồi thường của mỗi bên (công ty chứng khoán và khách hàng) BE
Lưu ý pháp lý: Nội dung điều khoản (trading methods, risk disclosure) do pháp chế soạn thảo. Hệ thống IT cung cấp API để lưu trữ, versioning, hiển thị và tracking acceptance — không thay thế tư vấn pháp lý.

Flow tổng quan

Quy trình đăng ký/kích hoạt dịch vụ với 3 bước chính: xác minh SĐT → hiển thị điều khoản → đồng ý điều khoản.

[Đăng ký/Kích hoạt] │ ├─ 1. Xác minh SĐT (3-Layer) │ ├─ Layer 1: Carrier API (GSMA Open Gateway) │ │ ├─ Number Verification → SIM ownership confirm │ │ └─ SIM Swap Check → fraud detection │ │ │ ├─ Layer 2: VNeID/RAR C06 (Phase 2) │ │ └─ App-to-app identity verification │ │ │ ├─ Layer 3: OTP + eKYC (Fallback) │ │ ├─ OTP + cross-check hồ sơ │ │ └─ eKYC (CCCD + face matching) │ │ │ └─ (Nếu thay đổi SĐT) → Biometric L2 + re-verify │ ├─ 2. Hiển thị Điều khoản dịch vụ │ ├─ Lấy nội dung T&C mới nhất (versioned) │ ├─ Phương thức GD online & loại GD tương ứng │ └─ Rủi ro & trách nhiệm bồi thường │ ├─ 3. Đồng ý/Chấp nhận điều khoản │ ├─ Accept → ghi nhận lịch sử │ └─ Reject → không thể tiếp tục sử dụng dịch vụ │ └─ 4. (Phase 2) Contract execution — lưu hợp đồng điện tử

03

Core Modules

Feature gồm 3 module chính, mỗi module giải quyết một yêu cầu cụ thể của Khoản 4 Điều 5.

BE+FE Phone Verification

Yêu cầu: Xác minh SĐT thuộc quyền sử dụng hợp pháp của KH (xác minh từ bên thứ ba).

Kiến trúc 3-Layer:

  • Layer 1 — Carrier API: GSMA Open Gateway CAMARA Number Verification + SIM Swap (Viettel, VinaPhone, MobiFone)
  • Layer 2 — VNeID/RAR: App-to-app identity verification qua C06 (Phase 2)
  • Layer 3 — OTP + eKYC: Fallback khi carrier/VNeID unavailable

BE POST /api/v1/phone/verify (carrier API primary)

BE POST /api/v1/phone/verify-otp (OTP fallback)

BE POST /api/v1/phone/change

BE+FE Terms & Conditions

Yêu cầu: Dịch vụ phải thể hiện bằng HĐ/điều khoản HĐ với KH.

Cơ chế:

  • Versioning terms (tạo, publish, deprecate)
  • Acceptance tracking (who, when, which version)
  • Grace period cho version mới (7 ngày)
  • Force accept: block giao dịch nếu chưa accept

BE GET /api/v1/terms/latest

BE POST /api/v1/terms/:id/accept

BE GET /api/v1/terms/status

BE+FE Trading Methods & Risk Disclosure

Yêu cầu: Hợp đồng phải bao gồm phương thức GD trực tuyến & loại GD tương ứng, và các rủi ro & trách nhiệm bồi thường.

Cơ chế:

  • Trading methods registry: Bảng cấu hình động (Mobile App, Web Trading, API) mapping với loại lệnh (Regular, Stop Limit, TP/SL, OCO)
  • Risk disclosure: Nội dung versioned do pháp chế quản lý, trả về qua API
  • Cache: 1h TTL (dữ liệu ít thay đổi)
  • Admin UI: Config trading methods, CRUD risk content (Phase 1 cơ bản)
BE GET /api/v1/trading-methods BE GET /api/v1/risk-disclosure

API Endpoints chính

Các API endpoints chính của feature. Xem chi tiết trong Markdown spec.

4.1 Phone Verification — Initiate

POST /api/v1/phone/verify

Initiate phone verification. BE tự động chọn method: Carrier API (primary) → OTP (fallback).

ParamTypeReqDescription
phoneCountryCodestring✅Mã quốc gia, e.g. "+84"
phoneNumberstring✅Số điện thoại
purposestring✅REGISTER / CHANGE / VERIFY
fullNamestring✅Dùng cho OTP fallback + cross-check
dobstring✅Ngày sinh (yyyy-MM-dd)
preferMethodstring❌CARRIER_API (default) / OTP
Response (Carrier): {"status":"carrier_confirmed","method":"CARRIER_API","carrierRefId":"...","operator":"VIETTEL","verificationResult":"CONFIRMED"}

4.2 Phone — Initiate Change

POST /api/v1/phone/change

Đổi SĐT đăng ký. Yêu cầu biometric L2 verification trước khi cho phép đổi.

ParamTypeReqDescription
newPhoneCountryCodestring✅Mã quốc gia mới
newPhoneNumberstring✅SĐT mới
fullNamestring✅Cross-check
dobstring✅Ngày sinh
biometricProofTokenstring✅JWT token từ biometric L2
Security: Bắt buộc biometric L2. Nếu không có → BIOMETRIC_REQUIRED. Gửi notification đến SĐT cũ khi thay đổi thành công.

4.3 Terms — Check Status

GET /api/v1/terms/status

Check xem user đã accept terms mới nhất chưa. Được gọi sau login để FE quyết định có show terms screen không.

{
  "hasPendingTerms": false,
  "latestVersionId": 5,
  "latestVersionTag": "v2.0",
  "acceptedVersionId": 5,
  "requiresAction": false
}

4.4 Terms — Accept / Reject

POST /api/v1/terms/:versionId/accept

Ghi nhận user đồng ý hoặc từ chối điều khoản.

{
  "action": "ACCEPTED",
  "deviceFingerprintId": 789
}

Terms Acceptance — UX Flow

Hai flow chính: lần đầu đăng ký và khi có version điều khoản mới.

6.1 First-time Registration

[Đăng ký tài khoản thành công] │ ├─ GET /terms/status → hasPendingTerms = true │ ├─ GET /terms/latest → Nội dung T&C ├─ GET /trading-methods → Danh sách phương thức GD └─ GET /risk-disclosure → Rủi ro & trách nhiệm │ ▼ TermsScreen hiển thị 3 tab: ├─ 📄 Điều khoản dịch vụ ├─ 📱 Phương thức giao dịch └─ ⚠️ Rủi ro & trách nhiệm │ ▼ [ "Tôi đã đọc và đồng ý" ] [ "Từ chối" ] │ │ ▼ ▼ POST /terms/:id/accept → Không thể sử dụng dịch vụ → Ghi nhận acceptance → Cho phép sử dụng dịch vụ

6.2 New Version Available

[Terms được admin publish version mới] │ ├─ User đăng nhập │ ├─ GET /terms/status │ ├─ hasPendingTerms = false → tiếp tục bình thường │ └─ hasPendingTerms = true → TermsRequiredScreen │ ├─ TermsRequiredScreen: │ ├─ Hiển thị diff (thay đổi so với version cũ) │ ├─ Grace period countdown (nếu còn) │ ├─ [ "Tôi đồng ý" ] / [ "Tìm hiểu thêm" ] │ └─ Nếu không accept trong grace period (7 ngày) → Block giao dịch
Grace Period: User có 7 ngày để đọc và đồng ý với phiên bản điều khoản mới. Trong thời gian này, giao dịch vẫn hoạt động bình thường. Hết grace period mà chưa accept → block giao dịch cho đến khi đồng ý.
07

Security & Risks

Các rủi ro bảo mật và biện pháp giảm thiểu cho Service Agreement feature.

#ConcernRiskMitigation
1 Carrier API outage Không verify được SĐT qua nhà mạng Fallback OTP + cross-check. Timeout 5s. Circuit breaker pattern
2 Carrier API fraud Giả mạo response từ carrier API key + IP whitelist + mutual TLS. Verify response signature
3 SIM swap fraud SĐT bị đổi SIM gần đây SIM Swap API check. Nếu phát hiện → yêu cầu OTP bổ sung
4 OTP brute force Kẻ tấn công thử nhiều mã OTP Rate limit 5 lần/giờ, max 5 attempts, exponential backoff
5 SĐT thuộc người khác KH đăng ký SĐT không thuộc quyền sở hữu Carrier API verification (primary). OTP + cross-check (fallback)
6 Phone change fraud Kẻ gian chiếm quyền đổi SĐT Carrier re-verify + biometric L2 + notification đến SĐT cũ
7 Terms acceptance forgery Giả mạo accept terms Device fingerprint + IP logging trong acceptance record
8 Content injection Nội dung terms bị sửa trái phép Admin-only write, input sanitization, phiên bản immutable sau publish
