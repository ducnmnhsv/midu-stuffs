Flow Overview

Session management hoạt động xuyên suốt từ login đến logout. User chỉ tương tác khi cần — xem danh sách thiết bị hoặc revoke session.

Bước 1
Login + Session check
Khi login, BE kiểm tra số lượng active sessions. Nếu đạt giới hạn (mặc định 3), từ chối với thông báo rõ ràng — không silent reject
BE + FE
Bước 2
Xem danh sách thiết bị
User vào Settings → "Quản lý thiết bị" để xem tất cả sessions đang active — không popup, không spam
FE
Bước 3
Revoke session từ xa
User chọn thiết bị cần revoke → swipe/xác nhận → BE soft-delete token — thiết bị kia bị logout ngay lập tức
BE + FE
Bước 4
Cảnh báo timeout
FE hiển thị banner trước 5 phút khi session sắp hết hạn — user không bị "bất ngờ" logout
FE

04⚙️

Concurrent Session Enforcement

Chính sách REJECT — user chủ động revoke thiết bị cũ trước khi login thiết bị mới.

Config BE

session.maxConcurrentSessions:
  DEFAULT: 3
  VIP:     5
  ADMIN:   10

session.warningThreshold: 1
session.deviceHistoryDays: 90

Policy Decision BE

Phase 1: REJECT — user kiểm soát chủ động. Khi đạt limit, từ chối login kèm hướng dẫn revoke.

Phase 2 (optional): FIFO — auto-revoke oldest inactive session. Dễ gây confusion cho user.

Khuyến nghị: giữ REJECT policy.

🔑 Edge cases: Refresh token refresh → KHÔNG đếm là session mới. Biometric re-login cùng device → KHÔNG đếm là session mới. Admin accounts → không bị giới hạn session limit.

08⏰

Session Timeout UX

Không để user bị "bất ngờ" mất phiên — có cảnh báo trước và tùy chọn kéo dài session.

Timeline

  • T-5 phút: FE check remainingMinutes → hiển thị toast/banner "Phiên đăng nhập sẽ hết hạn trong 5 phút" + nút "Kéo dài phiên"
  • T-0: Session hết hạn → alert + clear local state → navigate về LoginScreen
  • Background resume: Khi app mở lại, check remainingMinutes → xử lý tương ứng

FE Implementation FE

  • Timer: Tính từ sessionExpiresAt (API /sessions/timeout)
  • Check interval: 60 giây / khi app resume từ background
  • Extend: Gọi refresh token → cập nhật expired_at
  • Biometric auto-login: Silent refresh session nếu đã enroll biometric

Open Questions

Các quyết định cần PM/compliance confirm trước khi implement.

1
maxSessions mặc định: 3 hay 5?
Mặc định là 3 cho user thường. VIP user có thể được set 5. Admin không bị giới hạn. Con số này có thể điều chỉnh sau Phase 1.
Cần: PM + Compliance
2
Policy khi đạt limit: REJECT hay FIFO?
REJECT (Phase 1) — user chủ động revoke. FIFO auto-revoke session cũ nhất có thể gây confusion ("tại sao tôi bị logout?"). Nên giữ REJECT.
Quyết định: REJECT
3
Admin có bypass session limit không?
Có — admin operations không nên bị chặn bởi session limit. Admin có maxSessions riêng (10).
Quyết định: Admin không bị limit
4
Tính session limit gộp chung hay riêng theo login method?
Gộp chung — mọi login method (password, biometric, OTP) đều tính là 1 session. Tránh trường hợp user login N lần với N method khác nhau.
Quyết định: Gộp chung
